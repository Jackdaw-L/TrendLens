下面是一篇文章的原文材料，以及基于它的中文转写稿。请核查转写稿是否忠实于原文材料。

只有以下情况算「硬伤」，判 fail：
1. 编造：转写里出现原文材料中不存在的具体事实——数字、日期、人名、机构、产品功能、引语、案例、商业结论。
2. 错乱：数字/单位/百分比抄错，主体张冠李戴（把 A 公司的事安到 B 公司头上），因果或时间线颠倒。
3. 立场失真：原文谨慎或存疑的表述被转写成确定结论；原文的批评被写成认可（或反之）。
4. 无中生有的细节堆砌：原文材料很短，转写却大量补充原文没有的具体情节。

以下情况明确「不算问题」，不要因此判 fail：
- 中文重述、句式重组、段落结构调整、合理的比喻。
- 术语注释（annotations）里对公共常识概念的解释。
- whyRecommended / whyNow / pmAngle / pmTakeaways 中带 PM 视角的延伸判断（这些字段允许个人观点）。
- 正文中已明确标注「原文没明说，但可以推断」「从 PM 视角看」之类的延伸。
- 对原文次要信息的省略（除非省略导致误导）。
- 转写口吻直接陈述原文中的事实和观点（这是产品要求，不是编造）。

输出 JSON 结构（不要 Markdown 代码块）：
{
  "verdict": "pass" 或 "fail",
  "issues": [
    {
      "quote": "转写稿中有问题的原句（截取关键部分即可）",
      "problem": "一句话说明问题类型和错在哪",
      "evidence": "原文对应表述；若原文根本没有相关内容，写「原文无此内容」"
    }
  ]
}

规则：
- verdict 为 fail 时 issues 至少 1 条，最多 5 条，按严重程度排序。
- verdict 为 pass 时 issues 为空数组。
- 拿不准是否算硬伤时，倾向 pass——宁可放过轻微问题，不要用猜测打回一篇合格转写。
